# 双色球数据分析与智能推送系统接口设计与任务拆解

## 1. 文档目的
本文档在开发文档基础上进一步细化：

1. 模块接口定义。
2. 数据对象约定。
3. CLI 与可选 HTTP 接口设计。
4. 开发任务拆解。
5. 阶段验收标准。

本文档面向开发执行，目标是让项目可以直接按任务进入编码。

## 2. 开发原则

### 2.1 设计原则
1. 优先实现单机闭环，不做过早平台化设计。
2. 先 CLI、后 HTTP API。
3. 先 SQLite、后 MySQL。
4. 所有核心能力必须支持手动触发和定时触发两种执行方式。

### 2.2 编码约束
1. 业务逻辑与外部依赖解耦。
2. 各模块输入输出采用明确的数据对象，不直接传散乱字典。
3. 推送、爬虫、数据库访问均需支持替换实现。
4. 所有任务执行结果必须落日志表。

## 4. 领域对象设计

### 4.1 核心 DTO

#### 4.1.1 `DrawResultDTO`
```python
@dataclass
class DrawResultDTO:
    issue_no: str
    draw_date: str
    red_numbers: list[int]
    blue_number: int
    source_name: str
    source_url: str
    raw_payload: str | None = None
```

约束：

1. `red_numbers` 长度必须为 6。
2. `red_numbers` 必须升序且不重复。
3. `blue_number` 范围必须在 1-16。

#### 4.1.2 `AnalysisSnapshotDTO`
```python
@dataclass
class AnalysisSnapshotDTO:
    based_on_issue: str
    snapshot_date: str
    window_size: int
    red_frequency: dict[int, float]
    blue_frequency: dict[int, float]
    omission_map: dict[str, int]
    hot_numbers: list[int]
    cold_numbers: list[int]
    pattern_stats: dict
```

#### 4.1.3 `StrategyContext`
```python
@dataclass
class StrategyContext:
    snapshot: AnalysisSnapshotDTO
    config: dict
    strategy_params: dict
    rng_seed: int | None = None
```

#### 4.1.4 `RecommendationDTO`
```python
@dataclass
class RecommendationDTO:
    batch_no: str
    recommend_date: str
    target_issue_no: str | None
    strategy_name: str
    red_numbers: list[int]
    blue_number: int
    feature_summary: dict
    snapshot_id: int | None = None
```

#### 4.1.5 `CheckResultDTO`
```python
@dataclass
class CheckResultDTO:
    recommendation_id: int
    issue_no: str
    red_hits: int
    blue_hit: bool
    prize_level: str | None
    prize_amount: float | None
```

## 5. 模块接口设计

### 5.1 配置模块 `config`

#### 5.1.1 目标
负责读取 `config.json`、环境变量覆盖、默认值补齐与格式校验。

#### 5.1.2 接口
```python
class ConfigService:
    def load(self, path: str = "config.json") -> dict: ...
    def validate(self, config: dict) -> None: ...
    def get(self, key_path: str, default=None): ...
```

#### 5.1.3 验收标准
1. 缺失必要字段时抛出明确错误。
2. 时间、数值、渠道配置不合法时阻止启动。

### 5.2 数据采集模块 `collector`

#### 5.2.1 目标
负责抓取历史开奖数据和最新开奖数据。

#### 5.2.2 接口
```python
class DrawCollector:
    def fetch_history(self, start_date: str, end_date: str) -> list[DrawResultDTO]: ...
    def fetch_latest(self) -> DrawResultDTO | None: ...
```

```python
class DrawParser:
    def parse_history(self, html: str) -> list[DrawResultDTO]: ...
    def parse_latest(self, html: str) -> DrawResultDTO | None: ...
```

#### 5.2.3 关键规则
1. 抓取与解析拆分实现。
2. 采集失败时不得写入脏数据。
3. 重试后仍失败则记录 `job_runs`。

### 5.3 仓储模块 `repository`

#### 5.3.1 `DrawRepository`

class DrawRepository:

### 5.4 统计分析模块 `analyzer`

#### 5.4.1 目标
根据历史开奖数据生成统计快照。

#### 5.4.2 接口
```python
class AnalysisService:
    def build_snapshot(self, window_size: int | None = None) -> AnalysisSnapshotDTO: ...
```

#### 5.4.3 子能力接口
```python
class FrequencyAnalyzer:
    def calc_red_frequency(self, draws: list[DrawResultDTO]) -> dict[int, float]: ...
    def calc_blue_frequency(self, draws: list[DrawResultDTO]) -> dict[int, float]: ...
```

```python
class OmissionAnalyzer:
    def calc_omission(self, draws: list[DrawResultDTO]) -> dict[str, int]: ...
```

```python
class PatternAnalyzer:
    def calc_pattern_stats(self, draws: list[DrawResultDTO]) -> dict: ...
```

#### 5.4.4 输出要求
1. 能直接被策略模块消费。
2. 支持持久化为 JSON 字段。

### 5.5 策略模块 `strategy`

#### 5.5.1 基础接口
```python
class BaseStrategy(Protocol):
    name: str
    def generate(self, context: StrategyContext) -> RecommendationDTO: ...
```

#### 5.5.2 策略选择器接口
```python
class StrategySelector:
    def choose(self, enabled_strategies: list[dict]) -> BaseStrategy: ...
    def rebalance(self, recent_metrics: dict) -> list[dict]: ...
```

#### 5.5.3 规则
1. 每次只返回一个策略实例生成一组号码。
2. 5 组号码可由同一策略生成，也可由多个策略混合生成。
3. 选号后必须经过统一合法性校验。

### 5.6 推荐模块 `recommender`

#### 5.6.1 目标
负责生成一批推荐号码并落库。

#### 5.6.2 接口
```python
class RecommendationService:
    def generate_daily(self, count: int = 5) -> list[RecommendationDTO]: ...
    def resolve_target_issue(self) -> str | None: ...
```

#### 5.6.3 逻辑要求
1. 获取最新统计快照。
2. 按策略权重挑选策略。
3. 生成号码并过滤非法或重复组合。
4. 统一格式化后入库。

### 5.7 开奖核对模块 `checker`

#### 5.7.1 接口
```python
class PrizeRuleService:
    def resolve_prize(self, red_hits: int, blue_hit: bool) -> tuple[str | None, float | None]: ...
```

```python
class CheckService:
    def check_issue(self, issue_no: str) -> list[CheckResultDTO]: ...
    def compare_numbers(self, recommendation: RecommendationDTO, draw: DrawResultDTO) -> CheckResultDTO: ...
```

#### 5.7.2 验收标准
1. 奖级映射正确。
2. 同一期号重复执行时结果不重复插入。

### 5.8 推送模块 `notifier`

#### 5.8.1 基础接口
```python
class BaseNotifier(Protocol):
    channel_name: str
    def send(self, title: str, content: str) -> bool: ...
```

#### 5.8.2 格式化接口
```python
class MessageFormatter:
    def format_recommendation(self, items: list[RecommendationDTO]) -> tuple[str, str]: ...
    def format_check_result(self, issue_no: str, results: list[CheckResultDTO]) -> tuple[str, str]: ...
    def format_system_alert(self, message: str) -> tuple[str, str]: ...
```

#### 5.8.3 约束
1. 渠道发送失败不得影响主业务结果入库。
2. 必须保留请求结果用于追踪。

### 5.9 调度模块 `scheduler`

#### 5.9.1 接口
```python
class SchedulerService:
    def register_jobs(self) -> None: ...
    def run_forever(self) -> None: ...
```

#### 5.9.2 任务触发顺序
1. `sync_latest_draw_job`
2. `analyze_job`
3. `check_result_job`

抓取最新开奖成功后，后续两个任务应立即串联执行。

## 6. CLI 设计

### 6.1 命令清单

#### 6.1.1 初始化
```bash
python -m app init-db
```

行为：

1. 创建数据库文件。
2. 创建全部数据表。

#### 6.1.2 同步历史数据
```bash
python -m app sync-history --start 2021-01-01 --end 2026-05-23
```

行为：

1. 抓取区间数据。
2. 清洗并去重入库。
3. 输出新增数量。

#### 6.1.3 同步最新开奖
```bash
python -m app sync-latest
```

#### 6.1.4 生成统计
```bash
python -m app analyze --window 30
```

#### 6.1.5 生成推荐
```bash
python -m app recommend --count 5
```

#### 6.1.6 核对开奖
```bash
python -m app check --issue 2026052
```

#### 6.1.7 启动调度器
```bash
python -m app run-scheduler
```

### 6.2 返回规范
CLI 输出统一包括：

1. 执行状态。
2. 核心结果数量。
3. 失败原因摘要。

## 8. 开发任务拆解

### 8.1 阶段一：项目初始化

#### 8.1.1 任务
1. 初始化项目目录结构。
2. 建立 `requirements.txt`。
3. 建立 `config.json` 示例文件。
4. 建立基础日志能力。
5. 建立 CLI 主入口。

#### 8.1.2 产出
1. 可运行的空项目。
2. `python -m app --help` 可执行。

#### 8.1.3 验收标准
1. 项目能启动。
2. 配置文件可加载。
3. 日志能输出到控制台或文件。

### 8.2 阶段二：数据库与仓储层

#### 8.2.1 任务
1. 建立 SQLite 连接管理。
2. 创建全部数据表。
3. 实现各 Repository。
4. 补充基础 CRUD 测试。

#### 8.2.2 产出
1. `init-db` 命令可建库建表。
2. Repository 支持基本读写。

#### 8.2.3 验收标准
1. 期号唯一约束生效。
2. 推荐与核对记录能正确关联。

### 8.3 阶段三：数据采集

#### 8.3.1 任务
1. 实现官方数据源抓取器。
2. 实现 HTML 解析逻辑。
3. 实现数据清洗和合法性校验。
4. 接入历史同步和最新同步命令。

#### 8.3.2 产出
1. `sync-history` 可入库近五年数据。
2. `sync-latest` 可补最新一期。

#### 8.3.3 验收标准
1. 同步结果无重复期号。
2. 红球蓝球范围全部合法。

### 8.4 阶段四：统计分析

#### 8.4.1 任务
1. 实现频率统计。
2. 实现遗漏值计算。
3. 实现冷热号识别。
4. 实现形态统计。
5. 生成统计快照并入库。

#### 8.4.2 产出
1. `analyze` 命令可输出最新快照。

#### 8.4.3 验收标准
1. 统计结果可重复计算且稳定。
2. 快照可被策略模块直接消费。

### 8.5 阶段五：选号策略与推荐

#### 8.5.1 任务
1. 实现随机策略。
2. 实现冷热搭配策略。
3. 实现形态过滤策略。
4. 实现策略权重选择器。
5. 实现推荐批次生成与落库。

#### 8.5.2 产出
1. `recommend` 命令生成 5 组号码。
2. 推荐记录入库。

#### 8.5.3 验收标准
1. 每组号码满足双色球规则。
2. 同一批次内不重复。
3. 推荐记录带有策略来源。

### 8.6 阶段六：开奖核对

#### 8.6.1 任务
1. 实现红蓝球命中计算。
2. 实现奖级映射规则。
3. 实现指定期号推荐核对。
4. 核对结果入库。

#### 8.6.2 产出
1. `check` 命令可核对指定期号。

#### 8.6.3 验收标准
1. 奖级判断符合规则。
2. 重复核对同一期号保持幂等。

### 8.7 阶段七：推送能力

#### 8.7.1 任务
1. 接入首个推送渠道。
2. 实现推荐消息格式化。
3. 实现中奖消息格式化。
4. 落推送记录表。

#### 8.7.2 产出

#### 8.7.3 验收标准
1. 推送成功和失败均可追踪。

### 8.8 阶段八：定时调度与稳定性

#### 8.8.1 任务
1. 接入 APScheduler。
2. 注册每日推荐任务。
3. 注册开奖日同步与核对任务。

## 9. 测试任务拆解

### 9.1 单元测试优先级
1. `validators.py`
2. `prize_rules.py`
3. `omission.py`
4. `pattern.py`
5. `selector.py`

### 9.2 集成测试场景
1. 历史抓取到入库全链路。
2. 统计快照生成到推荐生成全链路。
3. 最新开奖同步到核对推送全链路。

### 9.3 回归重点
1. 期号重复写入。
2. 红球排序不一致导致核对异常。
3. 策略生成死循环。
4. 推送失败引发主任务中断。

## 10. 建议开发顺序

### 10.1 最短闭环顺序
1. 项目初始化。
2. 数据库与仓储层。
3. 历史抓取。
4. 统计分析。
5. 随机策略推荐。
6. 核对逻辑。
7. 单渠道推送。
8. 调度器。

### 10.2 原因
这个顺序能先跑通业务主链路，再逐步替换为多策略和更复杂的自动化逻辑，返工最少。

## 11. 人天估算建议

### 11.1 单人开发估算
1. 项目初始化与数据库：1-2 天。
2. 采集模块：2-3 天。
3. 分析模块：2 天。
4. 推荐与策略：2-3 天。
5. 核对模块：1 天。
6. 推送与调度：2 天。
7. 测试与修复：2-3 天。

合计建议：10-16 天。

## 13. 下一步建议
如果继续进入实现阶段，建议立即开始以下工作：

1. 初始化项目骨架。
2. 建立 SQLite 表结构与 Repository。
3. 先完成 `init-db`、`sync-history`、`analyze` 三个命令。
