# 双色球数据分析与智能推送系统开发文档

## 1. 文档目的
本文档基于《双色球数据分析与智能推送系统需求文档》输出可执行的开发方案，用于指导系统设计、编码实现、测试验收与后续迭代。

## 2. 需求分析结论

### 2.1 项目定位
本系统是一个面向单用户或小规模订阅用户的自动化分析与推送工具，核心价值在于：

1. 自动采集近五年的双色球历史开奖数据，并持续更新。
2. 基于历史统计特征生成每日推荐号码。
3. 在开奖后自动核对历史推荐结果并推送通知。

### 2.2 需求边界
需求文档中的“智能选号”“自动调整”不应实现为预测模型，而应实现为基于统计规则的号码生成与策略权重轮换机制。

系统不包含以下能力：

1. 不承诺提高中奖概率。
2. 不接入真实购彩、支付、兑奖流程。
3. 不实现复杂后台管理系统作为首期目标。
4. 不依赖高并发架构，首期按单机部署设计。

### 2.3 建议的首期交付范围（MVP）
首期版本建议聚焦以下能力：

1. 历史开奖数据初始化抓取。
2. 定时更新最新开奖数据。
3. 统计频率、冷热号、遗漏值、形态特征。
4. 生成 5 组推荐号码。
5. 通过至少 1 种推送渠道发送通知，优先支持 Server 酱或企业微信机器人。
6. 开奖后自动核对推荐记录并生成中奖结果。
7. 支持本地 `config.json` 配置。

以下内容作为二期：

1. 多渠道同时推送。
2. Web 管理后台。
3. 更复杂的策略评分体系。
4. Docker Compose + MySQL 部署方案。

## 3. 系统总体设计

### 3.1 架构选型
推荐采用 Python 单体应用 + SQLite + APScheduler 的轻量架构。

原因：

1. 需求以定时任务、数据抓取、统计计算为主，Python 生态合适。
2. 初期数据量小，SQLite 足够支撑。
3. 部署简单，适合本地或轻量云主机。

### 3.2 逻辑架构
系统划分为以下模块：

1. `collector`：数据采集与清洗。
2. `repository`：数据库访问层。
3. `analyzer`：统计分析计算。
4. `strategy`：选号策略与策略调度。
5. `recommender`：推荐号码生成。
6. `checker`：开奖比对与奖级计算。
7. `notifier`：推送渠道封装。
8. `scheduler`：定时任务编排。
9. `config`：配置读取与校验。
10. `app`：CLI 入口与应用启动。

### 3.3 核心业务流

#### 3.3.1 历史初始化流程
1. 读取数据源配置。
2. 抓取近五年历史开奖数据。
3. 清洗并标准化字段。
4. 去重后写入 `draw_results`。
5. 触发一次全量分析，生成统计快照。

#### 3.3.2 每日推荐流程
1. 读取最新统计结果。
2. 根据策略权重生成候选号码。
3. 使用形态过滤规则筛选。
4. 生成 5 组推荐记录。
5. 调用推送模块发送推荐消息。
6. 记录任务日志与推送结果。

#### 3.3.3 开奖核对流程
1. 在开奖后拉取最新开奖数据。
2. 查找待核对推荐记录。
3. 计算红球命中数、蓝球命中情况。
4. 根据奖级规则生成核对结果。
5. 推送中奖或未中奖通知。

## 4. 模块详细设计

### 4.1 数据采集模块

#### 4.1.1 输入
1. 数据源 URL。
2. 抓取时间范围。
3. 请求重试与超时配置。

#### 4.1.2 输出
标准化后的开奖数据：

- 期号
- 开奖日期
- 红球 6 个
- 蓝球 1 个
- 原始来源信息

#### 4.1.3 实现要求
1. 优先使用 `requests + BeautifulSoup`。
2. 页面结构不稳定时，再引入 `Selenium` 作为备选方案。
3. 每条数据按 `issue_no` 唯一约束去重。
4. 写库前校验红球数量、范围、去重性，蓝球范围。
5. 保留原始响应内容或摘要，便于排查数据问题。

### 4.2 统计分析模块

#### 4.2.1 统计指标
1. 红球 1-33 的出现次数与出现概率。
2. 蓝球 1-16 的出现次数与出现概率。
3. 每个号码当前遗漏值。
4. 热号、冷号集合。
5. 奇偶比统计。
6. 大小比统计。
7. 三区分布统计。
8. 红球和值分布统计。

#### 4.2.2 指标定义建议
1. 热号：最近 `N` 期内频次位于前 30% 的号码。
2. 冷号：最近 `N` 期内频次位于后 30% 的号码。
3. 遗漏值：距最近一次开出已间隔的期数。
4. 大小比：红球按 `1-16` 为小、`17-33` 为大。
5. 三区比：红球按 `1-11`、`12-22`、`23-33` 分区。

`N` 不应写死，放入配置文件，默认建议为 30 期。

### 4.3 选号策略模块

#### 4.3.1 策略列表
1. `random_strategy`：纯随机生成。
2. `hot_cold_strategy`：按冷热比例抽取。
3. `pattern_filter_strategy`：先生成，再按形态规则过滤。

#### 4.3.2 策略接口约定
每个策略实现统一接口：

```python
class Strategy:
    name: str

    def generate(self, context: StrategyContext) -> Recommendation:
        ...
```

`StrategyContext` 应包含：

1. 最新统计数据。
2. 配置参数。
3. 当前策略权重。
4. 随机种子或随机数生成器。

#### 4.3.3 自动调整逻辑
建议实现为“策略权重重算”，而不是“预测修正”。

规则示例：

1. 每累计 100 期执行一次权重调整。
2. 统计最近 100 期中，各策略若按模拟规则生成号码时的平均命中表现。
3. 将表现较优策略权重上调，较差策略下调。
4. 设置上下限，避免某一策略权重归零。

首期可简化为：

1. 按固定权重轮询策略。
2. 每 100 期根据冷热号开出比例调整冷热搭配策略的参数。

### 4.4 推荐生成模块
1. 每日生成固定 5 组号码。
2. 每组号码必须满足基础规则：6 红不重复、1 蓝。
3. 同一批次内推荐号码应避免完全重复。
4. 推荐记录需保存生成来源策略、使用参数、统计快照版本。

### 4.5 开奖核对模块

#### 4.5.1 规则说明
该模块仅执行“中奖核对”，不执行真实兑奖。

#### 4.5.2 处理逻辑
1. 读取指定期号的开奖号码。
2. 查询该期对应的推荐记录。
3. 统计红球命中数与蓝球命中状态。
4. 按双色球奖级规则映射奖项。
5. 若配置了奖金表，则同步返回奖金金额。

#### 4.5.3 奖级规则建议
将奖级规则做成可配置映射，避免硬编码在业务逻辑中。

## 7. 定时任务设计

### 7.1 任务列表
1. `sync_history_job`：首次初始化或手动补数。
2. `sync_latest_draw_job`：开奖日抓取最新期数据。
3. `analyze_job`：更新统计快照。
4. `generate_recommendation_job`：每日生成推荐。
5. `check_result_job`：开奖后核对推荐结果。
6. `cleanup_job`：清理过期日志与临时数据。

### 7.2 调度建议
1. 每天早上固定时间生成推荐。
2. 每周二、四、日晚上开奖后抓取开奖数据。
3. 抓取成功后自动触发统计与核对任务。
4. 所有任务支持 CLI 手动执行，便于排查。

## 8. 接口与命令设计

### 8.1 CLI 命令
建议优先实现命令行入口，而不是先做 Web API。

```bash
python -m app init-db
python -m app sync-history
python -m app sync-latest
python -m app analyze
python -m app recommend
python -m app check
python -m app run-scheduler
```

### 8.2 可选 HTTP API（二期）
若后续需要接入后台，可增加以下接口：

1. `GET /api/draws/latest`
2. `GET /api/analysis/latest`
3. `GET /api/recommendations?date=YYYY-MM-DD`
4. `POST /api/tasks/recommend`
5. `POST /api/tasks/check`

## 10. 关键实现约束

### 10.1 数据质量
1. 同一期号不能重复入库。
2. 红球必须升序存储，便于比对与展示。
3. 抓取结果异常时，不允许覆盖已有正确数据。

### 10.2 容错要求
1. 外部数据源不可用时重试。
2. 多次失败后保留失败日志并发送系统告警。
3. 推送失败不应影响推荐记录入库。

### 10.3 可维护性
1. 各策略独立实现，便于扩展。
2. 奖级规则、形态阈值、冷热定义全部参数化。
3. 所有定时任务都应支持幂等执行。

## 11. 测试方案

### 11.1 单元测试
1. 号码合法性校验。
2. 遗漏值计算。
3. 冷热号识别。
4. 形态过滤规则。
5. 奖级判定逻辑。

### 11.2 集成测试
1. 抓取并入库完整流程。
2. 推荐生成并推送流程。
3. 开奖更新后自动核对流程。

### 11.3 异常测试
1. 数据源返回空数据。
2. 页面结构变更导致解析失败。
3. 推送接口超时或返回错误。
4. 数据库文件被占用或写入失败。

## 12. 部署方案

### 12.1 本地部署
1. Windows 使用任务计划程序。
2. Linux 使用 `cron` 或 `systemd`。

### 12.2 Docker 部署
首期只需单容器：

1. 应用容器内运行 Python 服务或调度器。
2. SQLite 数据文件挂载到宿主机卷。
3. 日志目录挂载到宿主机。

## 13. 开发里程碑建议

### 13.1 第一阶段：基础能力
1. 初始化项目结构。
2. 建表与数据库访问层。
3. 历史数据抓取与清洗。

### 13.2 第二阶段：分析与推荐
1. 统计分析模块。
2. 三类选号策略。
3. 推荐记录生成。

### 13.3 第三阶段：自动化闭环
1. 定时任务调度。
2. 推送渠道接入。
3. 开奖核对与通知。

### 13.4 第四阶段：稳定性完善
1. 单元测试与集成测试。
2. 日志、监控、错误恢复。
3. Docker 化部署。

## 14. 风险与待确认事项

### 14.1 风险
1. 官方数据源页面结构可能变化，导致解析逻辑失效。
2. 开奖发布时间可能波动，定时任务时间需要允许延迟与重试。
3. 部分推送渠道存在频率限制或鉴权变更风险。

### 14.2 待确认事项
1. 首期默认推送渠道具体选哪一个。
2. 是否需要多用户订阅能力。
3. 是否需要 MySQL 作为正式环境默认数据库。
4. 推荐号码是“每日推送”还是“仅开奖日前推送”。
5. 中奖金额是否维护静态奖级表，还是仅显示奖级不显示金额。

## 15. 结论
该项目适合按“单机定时任务系统”方式落地。首期不应过度设计为预测系统或复杂平台，应优先完成“抓取 - 分析 - 推荐 - 推送 - 核对”的业务闭环，在数据稳定后再逐步扩展策略与管理能力。
